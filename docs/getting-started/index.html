Getting Started

By Alex Rodriguez

Aura Docs provides a step-by-step guide for PAM on-boarding projects. The guide focuses on a workflow. A goal of the documentation is to define a process that integrates a customer's digital platform with the Aura sub-system. This guide is divided into five sections: 1) Overview, 2) Workflow, 3) Developer, 4) What's New, and 5) FAQ.

PAM Designer Workflow

The Privileged Access Management (PAM) Designer Workflow starts with the discovery of a Digital Platform. In this step, a platform is examined from the perspective of the access required by an MSSP to provide services to customers.

What is a Digital Platform?

It's a broader concept than machine platform in the sense that a Digital Platform is any environment that hosts workloads for customers. Digital platform includes machines, devices, applications, OSes, cloud-hosted resources, and other customer assets that are exposed as manageable PAM endpoints. Digital platforms include software and technology used in business operations and IT systems. A digital platform often serves as a company's backbone for operations and customer engagement. Services, solutions, tools, systems and interfaces comprise a digital platform.

What is AURA?

AURA stands for Access to User Resources Architecture, an integrated system that provides privileged access management for digital platforms. As an integration of cooperating components, a common goal of the system is to provision, monitor, and audit the use of elevated credentials. AURA includes a highly secure credentials vault to store credentials that grant the access needed by an MSSP to perform routine, authorized activities in support of business-critical operations, in a secure manner.

Credentials Manager - A web application, provides check-in/check-out. Designed for SOC Analysts.

Core APIs - Microservices to vault and retrieve credentials via REST.

Secret Server - An easy-to-use, full-featured PAM solution from Thycotic.

Credentials Vault - A highly scalable, encrypted storage facility included with Secret Server.

Workflow Steps

Discover

Step one in the worklow is Discover, an assessment of a digital platform focused on credentials to manage. Discovery creates or gains access to a Test Environment, an instance of the digital platform.

Design

The Design stage of the workflow designs secret templates and remote password changers (RPCs). During design, decisions are made about how to represent a credential to manage, and how to change passwords remotely.

Implement

The Implement step takes design input to create secret templates, remote password changers, as well as secrets (credentials) in the PAM system (Secret Server). Implementation uses the AURA development environment.

Validate

In the Validate stage of the workflow, secret templates are exported from a development environment and then imported into a staging environment for end-to-end testing and verification.

Deploy

The Deploy stage goes live by moving credentials into a production environment for authorized use by applications and users. Once credentials from a customer's digital platform are moved into production, and configured based on access requirements, then the PAM on-boarding workflow is complete.

Credential Categories

Credentials in AURA can be distinguished by category. One of the main differences between credential categories is the expected consumer. While static and functional credentials are mostly consumed by applications for integration and automation purposes, dynamic and controlled credentials are expected to be consumed by people for administrative purposes. Functional, dynamic, and controlled credentials are rotated by RPCs but on different intervals. Dynamic and controlled credentials rotate upon use (upon check-in), and functional credentials rotate yearly, automatically when they expire. Static credentials are not expected to change.

A credential from a digital platform to be managed by AURA, must belong to one of the categories:

Adapt It

There are many different ways of approaching new projects. This is a proposed PAM process, but it encourages teams to think about and tailor the process into something that works for them. There is no magic one-size-fits-all way to get through a project and achieve awesome results every time. Good design processes are both strong and flexible. This one is intended to allow teams to take on any PAM on-boarding challenge and to increase their chances of success by using the workflow steps as guides, not exact recipes.
