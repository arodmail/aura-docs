What's the difference between Secret and Credential?

Nothing. These are synonymous. The term secret comes up in the context of Thycotic Secret Server. Secrets in this product are individually named sets of sensitive information derived from secret templates. Flexibility in templates allows secrets to represent a very broad spectrum of secure data. In Secret Server, all secret field information is securely and individually encrypted.

The term credential is used interchangeably with secret, but a credential is narrower in scope than a secret in the sense that a credential is a form of identity that provides authentication and authorization into a digital platform. Examples of credentials include usernames and passwords, API keys, application tokens, SSH keys, SSL certificates, etc. As such, a credential is a kind of secret because it is modeled in Secret Server through a secret template.

In AURA, Credentials API performs the translation between the two concepts, because on the way out (GET), it retrieves secrets from Secret Server and uses the secure data returned to form and to represent a credential in a response. On the way in (POST), Credentials API accepts credential data and converts it into a secret to be vaulted securely in Secret Server.

Direct access to the Secret Server Web UI is restricted to authorized group members only.

Primarily, The Secret Server Web UI is intended to be used by Deployment Engineers who are authorized in the SS_SIE_Global AD Group for machine platform on-boarding, to create Secret Templates, configure Remote Password Changers, enable password rotation and lifecycle management, as well as to on-board customer devices into Secret Server by creating and/or discovering elevated credentials on managed devices.

On-boarding projects require the participation of an authorized Secret Server user. As such, contact a Deployment Engineer, someone in the SS_SIE_Global AD group, to step through the PAM Designer Workflow for your project.

How is a credential in AURA associated with a specific Customer? or Device? or Digital Platform?

Credentials in Aura are always associated with a Customer ID and/or Device ID. As such, they can be matched up using one or the other or both as parameters to retrieve credentials through Credentials API. For example, static device credentials are based upon a standard secret template that includes a field to specify the CustomerId, DeviceId, as well as Machine Platform, e.g., QRadar, Resilient, etc.

See the Design step of the PAM Designer Workflow for a list of standard secret template fields.
