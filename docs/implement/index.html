Implement

By Alex Rodriguez

The implementation step of the PAM on-boarding workflow creates secret templates, remote password changers, as well as secrets (credentials) in the PAM system (Secret Server) and runs implementation tests.

Implementation uses endpoints in the Development Environment for AURA:

EnvironmentEndpoint - LB URL
DevSecret Server
DevCredentials Manager
DevCredentials API

If LB URLs are not responsive, then Direct URLs may be used as alternatives to complete implementation tasks:

EnvironmentEndpoint - Direct URL
DevSecret Server
DevCredentials Manager
DevCredentials API

1. Create New Secret Template

Given a design for a secret template, implement the design, by creating the template and configuring it:

1. Login with your AD username/password to Secret Server - Dev

2. Select Admin > Secret Templates

3. If editing an existing template:

3.1 Select that template in the dropdown list

3.2 Click the Edit button to load it into the Secret Template Designer page.

4. If creating a new template:

4.1 Click the Create New button

4.2 Type the name of the new template in the text box: MSS + [Digital Platform Name]

4.3 Click the Create button, the Secret Template Designer page will appear

5. Add template fields and field types, based upon the secret template design.

6. Click the Edit button to customize template settings:

6.1 For Dynamic Credentials:

6.1.1 Expiration Enabled: Yes / Days = 75 / Change Required On: Password

6.1.2 Validate Password Requirement On Create?: Yes

6.1.3 Validate Password Requirement On Edit?: Yes

6.2 For Static Credentials:

6.2.1 Expiration Enabled: No

6.2.2 Validate Password Requirement On Create?: No

6.2.3 Validate Password Requirement On Edit?: No

6.3 For Controlled Credentials:

6.3.1 Expiration Enabled: Yes

6.3.2 Validate Password Requirement On Create?: Yes

6.3.3 Validate Password Requirement On Edit?: Yes

6.4 For Functional Credentials:

6.4.1 Expiration Enabled: Yes / Days = 365 / Change Required On: Password

6.4.2 Validate Password Requirement On Create?: No

6.4.3 Validate Password Requirement On Edit?: No

7. Click the Save button.

2. Create New Password Changer

If the design step provides for a custom password changer to be created, then follow the procedure below. Otherwise, to use a built-in RPC, skip this step and go to "Assign Password Changer to Secret Template".

See Thycotic Documentation for Creating a Custom Password Changer

3. Assign Password Changer to Secret Template

With a secret template created, it's ready to be assigned to an RPC, if required. Only Dynamic and Functional credentials require a remote password changer. This step may be skipped for Static and Controlled credential templates.

See Thycotic Documentation for Assigning a Password Changer to a Secret Template

4. Create Secrets (Credentials)

Refer to the Access Requirements from the Discover step in the PAM on-boarding worklow, repeat the following steps for each of the credentials required to be on-boarded from the digital platform, application, OS, host, device or cloud-hosted resource:

1. Click the plus sign in the upper right hand corner of the Web UI > Add Secret

2. Expand a parent folder in which to find a path for the secret. The following parent folders are available for each credential category:

  • Controlled

  • Dynamic

  • Static

  • Functional

3. Within a parent folder, define a path for the secret:

If a folder does not exist for [Customer ID] or [Digital Platform] or [Device ID]:

Go to Admin > Folders, expand the parent folder and create the missing folder(s) within.

3.1 Dynamic > Customers > [Customer ID] > Device > [Digital Platform] > [Device ID]

3.2 Static

3.2.1 Static > Customers > [Customer ID] > (Customer-level static secrets)

3.2.2 Static > Customers > [Customer ID] > Device > [Digital Platform] > [Device ID] > (Device-level static secrets)

3.3 Controlled > Customers > [Customer ID] > Device > [Digital Platform] > [Device ID]

3.4 Functional

3.4.1 Static > Customers > [Customer ID] > (Customer-level functional secrets)

3.4.2 Static > Customers > [Customer ID] > Device > [Digital Platform] > [Device ID] > (Device-level functional secrets)

4. Choose a Secret Template

5. Fill out all fields for the secret template selected, then click Create Secret

6. For dynamic and functional credentials check Auto Change Enabled

7. Go back to Step 1 and repeat as needed, per secret to be on-boarded.

5. Configure Check Out

For Dynamic and Controlled secrets, each secret must be individually set to require check out:

1. From the Secret View page, open the Security tab to modify a secretâ€™s Check Out setting.

2. Enable Require Check Out to force users to check out the secret before gaining access.

3. Enable Change Password on Check In to have the password change after the secret is checked in.

6. Request Network Access to Test Environment

Only Dynamic and Functional credentials require a remote password changer and network connectivity to a remote host. This step may be skipped for Static and Controlled credential templates. To request network access from the Secret Server to the Test Environment, submit a request in Remedy (or ServiceNow) for firewall rules to be created that accept/allow traffic from a Source to a Destination IP, where Source is Secret Server Development and Stage hosts and Destination is the IP address of a host in the Test Environment. 7. Implementation Test

The previous steps setup for implementation tests to be successful. The following tests can be followed for Dynamic, Static, Controlled and/or Functional credentials, depending on the credentials being on-boarded.

A. Secret Server

To test Dynamic and Controlled secrets/credentials through Secret Server:

1. Login with your AD username/password to Secret Server - Dev

2. Run a Heartbeat for each secret.

3. Check-out the secret

4. Confirm login/access to the managed endpoint using the password checked-out

5. Check-in the secret

6. If applicable, confirm password rotation is successful after check-in

Note: Heartbeat is only required to be run for Dynamic and Functional secrets, not required for Static or Controlled secrets.

B. Credentials Manager

To test Dynamic secrets/credentials through Credentials Manager:

1. Login with your AD username/password to Credentials Manager - Dev

2. Search for the credential

3. Check-out the credential

4. Confirm Login to the managed endpoint using the password checked-out

5. Check-in the credential

6. Confirm password rotation is successful after check-in

C. Credentials API

To test Static and/or Functional credentials access through Credentials API:

1. Login with your AD username/password to Credentials API - Dev

2. Search for the credential by search criteria:

2.1 Expand GET /credentials

2.2 Click Try it out

2.3 Select checkOutEnabled = false

2.4 Enter a templateName | machinePlatform | folderName | searchText

2.5 Click Execute

2.6 Confirm expected credential(s) are returned in the response based upon search criteria

2. Search for the credential by Device Id:

2.1 Expand GET /credentials/{deviceId}

2.2 Click Try it out

2.3 Enter a deviceId value

2.4 Click Execute

2.5 Confirm expected credential(s) are returned in the response based upon Device Id

9. Summary

The following 5 outcomes are expected from implementation tasks:
  1. Create New Secret Templates for Dynamic, Static, Controlled and/or Functional credentials
  2. Create New Password Changers, but only for Dynamic and Functional credentials
  3. Create Secrets (Credentials), as needed per Access Requirements identified during Discovery
  4. Request Network Access to Test Environment
  5. Run implementation tests to confirm access to credentials as well as check-out/in, if applicable
