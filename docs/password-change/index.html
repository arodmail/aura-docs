Password Change

By Alex Rodriguez

To allow privileged access for analysts and other users, Credentials Manager provides a web user interface to search, sort, and check-out/check-in dynamic and controlled credentials. Credentials can be used securely by analysts to access a customer's device, asset, application, operating system, cloud-hosted workload, or other digital platform resource.

Both dynamic and controlled credentials are enabled for check-out, but only dynamic credentials support automated password rotation through an RPC upon check-in or expiration. To satisfy compliance requirements, controlled credentials must change after each use, upon check-in or expiration, but are changed on a managed endpoint through a manual procedure.

Automated Password Change

When the password value of a dynamic credential expires, it changes automatically by a Remote Password Changer (RPC). It also changes when it is checked in. As these events happen, a transition to a different, strong password value defines a password lifecycle. Manual Password Change

For passwords that cannot be changed automatically, through an RPC or some equivalent automated change mechanism, a process to change passwords manually is available. Credentials Manager exposes both dynamic and controlled credentials to analysts, but the process by which controlled credentials are updated manually requires a Change Request Ticket and human intervention. The following diagram depicts the step-by-step flow for controlled credential check-out, use of a password value, followed by manual password change by a DevMon team member upon check-in.

Manual Change Process Flow

When Credentials API receives a request to check-in a controlled credential:

  1. A strong password value is generated
  2. The password value of the credential is updated with a new value
  3. The credential is checked-in, but only partially, "Pending" successful update on the endpoint*
  4. Creates a Change Request Ticket for the DevMon team to update the endpoint
  5. An audit log is created and retained to capture who checked-in the credential, when, and for what endpoint.

A Change Request Ticket created is routed to the DevMon team for manual password change:

  1. DevMon team member logs into Credentials Manager to complete the check-in process
  2. Credentials Manager only shows "Pending" credentials to DevMon team members (SS_DEVMON AD Group)
  3. Both previous and next password values are displayed by Credentials Manager, used to update the managed endpoint
  4. The "Pending" credential is checked-in through Credentials Manager and its status changes to "Available".
  5. The Change Request Ticket is Closed

Password Reset

For passwords that cannot be reset automatically prior to expiration, through an RPC or some equivalent automated change mechanism, a process to reset passwords manually is available. The following diagram depicts the step-by-step flow for controlled credential reset, that starts with a Change Request Ticket, followed by manual password reset by a DevMon team member.

Manual Reset Process Flow

When a credentials expires it appears under the "Expired" tab in Credentials Manager:

  1. A Change Request Ticket is submitted for the DevMon team to reset the password on the endpoint
  2. A strong password value is generated
  3. The "next" password value of the credential is updated with a new value

A Change Request Ticket created is routed to the DevMon team for manual password reset:

  1. DevMon team member logs into Credentials Manager to complete the reset process
  2. Credentials Manager only shows "Reset" credentials to DevMon team members (SS_DEVMON AD Group)
  3. Both previous and next password values are displayed by Credentials Manager, used to update the managed endpoint
  4. The "Reset" credential is reset through Credentials Manager and its status changes to "Available".
  5. An audit log is created and retained to capture who reset the credential, when, and for what endpoint.
  6. The Change Request Ticket is Closed
